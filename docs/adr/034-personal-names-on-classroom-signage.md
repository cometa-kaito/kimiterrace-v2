# ADR-034: 公開サイネージ（教室端末）への個人名表示（来校者一覧・生徒呼び出し）の許容と境界

- 状態: Accepted（2026-06-10、ユーザー判断）
- 日付: 2026-06-10（Proposed / Accepted 同日。パターン2 盤面の来校者一覧 #804/#805・生徒呼び出し 実装と同時着地）
- 関連: [[project_school_dx_no_teacher_burden]]（設計の最上位軸）, [ADR-030 authoring 時 PII ゲート](030-authoring-time-pii-gate.md), [ADR-016 class magic link 匿名アクセス](016-class-magic-link-anonymous-access.md), [ADR-003 Identity Platform](003-identity-platform.md), [CLAUDE.md ルール4（PII の Vertex 送信前マスキング）], パターン2 来校者一覧（class_visitors, [PR #804](https://github.com/cometa-kaito/kimiterrace-v2/pull/804) / [#805](https://github.com/cometa-kaito/kimiterrace-v2/pull/805)）, 生徒呼び出し（student_callouts, 本スライス）

## 文脈

パターン2 のサイネージ盤面に、運営要望で **個人名を含む運用情報**を表示する:

- **来校者一覧**: 来校者の氏名・所属・用件・対応者（外部の成人）。
- **生徒呼び出し**: 呼び出す生徒の **フルネーム** + 呼び出し先・用件・時刻（2026-06-10 ユーザー確定「フルネームで表示」）。

これは本プロジェクトの既存方針と**正面から衝突**する:

1. **設計の最上位軸**（[[project_school_dx_no_teacher_burden]]）: 「サイネージ＝クラス共通の公開情報のみ。個別生徒の氏名・出欠等の PII は出さない」。
2. **ADR-030 のコンテンツポリシー**: 「公開掲示物に個別生徒/保護者の氏名・特定可能情報を載せない」（authoring 時 soft-gate で警告）。
3. 生徒は **匿名設計**（[ADR-016]/[ADR-003]: 個別アカウント非保有・magic link 匿名アクセス、正規氏名 roster は構造的に不在）。

一方で、来校者掲示・生徒呼び出しは**実名でなければ運用が成立しない**（「出席番号 12 番、職員室へ」は取り違えやすく、来校者一覧から氏名を抜くと用をなさない）。ユーザーはこのトレードオフを承知の上で実名表示を選択した。

### 重要な区別（ルール4 とは別問題）

ADR-030 / ルール4 が守るのは **AI/embedding（Vertex）への PII 焼き込み**である。来校者・生徒呼び出しは:

- **構造化された運用データ**を盤面に**直接描画**するだけで、`content_versions` にも embedding にも入らず、**Vertex/LLM には一切送られない**。→ **ルール4（マスキング）は発火しない**。ADR-030 の content/AI 経路ゲートも非接触。
- 反転するのは **「公開サイネージ表示面に個別 PII を出さない」という設計軸**の方であり、これを運用機能のために**意識的に緩和**する判断が本 ADR の主題。

## 候補

| 候補 | 概要 | 評価 |
|---|---|---|
| A. 匿名化表示 | 出席番号 / イニシャルで呼び出し、来校者は氏名を伏せる | プライバシーは守るが**運用が成立しない**（呼び出しの取り違え・来校者一覧が無意味）。ユーザー要望（フルネーム）と不一致 |
| **B. 実名表示 + 境界づけ（採用）** | 氏名を表示するが、表示先・スコープ・経路・監査・保持を厳格に限定 | 運用を満たしつつ露出を構造的に最小化 |
| C. 機能を作らない | 来校者/呼び出しウィジェットを諦める | ユーザー要望を満たさない |

## 決定

**B（実名表示 + 境界づけ）** を採用する。設計の最上位軸を、**この 2 運用機能に限り**次の境界の下で緩和する:

1. **表示先の限定（classToken ゲート）**: 個人名を含む盤面は `/signage/{classToken}` でのみ描画される。`classToken` は credential（magic link、middleware 除外の匿名公開経路）であり、当該クラスの端末に配布されたトークンを持つ者だけが閲覧する。**インターネット全体への無条件公開ではない**。ただし**トークンが漏れれば氏名も漏れる**（物理的な教室内限定ではない）点は正直に認める＝下記「残存リスク」。
2. **RLS 自校スコープ（ルール2）**: `class_visitors` / `student_callouts` とも tenant_isolation で自校に限定。他校・他クラスからは構造的に不可視。
3. **Vertex 非送信（ルール4 不接触）**: これらの氏名は AI 抽出・embedding・RAG のいずれの経路にも入れない。盤面の payload で直接返すのみ。
4. **監査（ルール1）**: 編集は `audit_log`（tableName=class_visitors / student_callouts）に記録。ただし **diff に氏名そのものは焼かず**、対象クラス・日付・件数のみ（監査ログへの PII 二次蓄積を避ける。実体は本テーブルに残り RLS + 監査列で追跡可能）。
5. **最小保持・職員 curate**: いずれも **クラス×日別**のレコードで、**職員が明示的に入力**する（AI 自動生成・自動取込みではない）。当日の運用が終われば次の編集で置換/クリアされる（全置換 UI）。長期に生徒 PII を蓄積する設計ではない。
6. **生徒呼び出しの実名**: 呼び出し精度（取り違え防止）を理由にフルネームを表示する。表示は「呼び出し先・用件・時刻」と併記し、呼び出しの運用文脈に限定する。

### なぜ匿名化（A）でなく実名（B）か

呼び出しは**個人を一意に・即座に特定**できることが機能要件（番号呼び出しは取り違え・聞き取りミスを招く）。来校者一覧は**誰が来ているか**が本質で氏名を抜くと無意味。匿名化は機能を壊す。代わりに「**誰が見られるか（classToken + RLS）・どこへ流れるか（Vertex 非送信）・どれだけ残るか（日別・職員 curate）**」を締めることで露出を最小化する。

## 残存リスク（正直に明記）

- **classToken 漏洩**: トークンが共有/スクショ流出すると、その classToken の盤面（氏名含む）が外部から見えうる。保護は物理的教室限定ではなく**トークン credential + RLS**である。→ token の取り扱い（ログ非出力・失効運用）は既存方針（ルール5 / ADR-016）に従う。
- **盤面の物理的視認**: 教室 TV を通りがかりの第三者が見る可能性は元から存在（クラス共通情報も同様）。氏名表示はこの可視面に個人名を載せる点で露出を増やす。
- これらを許容するのが本 ADR のユーザー判断。

## 再検討トリガ

- サイネージがインターネット公開／教室外へ広く共有される運用へ変わった。
- 保護者・生徒・教育委員会等からプライバシー懸念・苦情が出た。
- 個人情報保護の法令/ガイドライン要件が変わった。
- 生徒/来校者 PII の保持期間が運用上問題化した → 古い callout/visitor 行の**自動クリーンアップ Job** を追加（本 ADR では未実装、トリガ時に検討）。

## 影響

- **設計軸 [[project_school_dx_no_teacher_burden]]**: 「サイネージに個別 PII を出さない」原則に、**来校者一覧・生徒呼び出しの 2 運用機能の例外**を本 ADR の境界条件付きで設ける（memory 側も追記）。
- **ADR-030 のコンテンツ/AI 経路ポリシーは不変**: 掲示物本文・AI 抽出に生徒氏名を載せない soft-gate はそのまま。本 ADR は構造化運用データの**表示面**の話で経路が別。
- **ルール4（Vertex マスキング）非接触**: 当該データを LLM/embedding に送らないことを実装で担保（payload 直返し）。
- スキーマ: `class_visitors`（既存）+ `student_callouts`（新規）。いずれも RLS + 監査 + Vertex 非送信。
