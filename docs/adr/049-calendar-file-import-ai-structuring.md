# ADR-049: 年間行事予定表ファイル（Excel/CSV 等）の AI 構造化取込（school_calendar_events 再利用・エディタ確定）

- 状態: Accepted（2026-07-12。Proposed 2026-07-11、レビュー #1261 通過・実コード整合検証済み）
- 日付: 2026-07-11
- 関連: [ADR-045 学校行事カレンダーの公開 iCal/ICS 取込（保存先テーブルの出自）](045-signage-school-calendar-ical-ingestion.md), [ADR-038 Gemini マルチモーダル OCR], [ADR-040 生徒 Q&A 知識源], [CLAUDE.md ルール1（監査）/ ルール2（RLS）/ ルール3（型単一ソース）/ ルール4（PII マスキング）], issue #1259

## 文脈

教員フィードバック（2026-07-11）: 「Excel や CSV（年間行事予定表などを想定）を読み込める入力モードが欲しい。フォーマットは学校ごとにバラバラなので AI に読み込ませるのが良い。今の AI 機能は短期間は良いが長期間の入力がしづらい」。

現状の構造的ギャップ（調査済み）:

1. **ファイル添付の口は既にある**（エディタ AI チャットの `assistDraftAllFromFileAction`。xlsx=exceljs / csv=UTF-8 / pdf / 画像 OCR の抽出レイヤ配線済み）が、出力先が**「編集中の 1 日分の予定/連絡/提出物への分類」に固定**されている。
2. 会話 AI の複数日下書きは `MAX_DRAFT_DAYS = 7`（`assistant-chat-core.ts`）がハード上限。年間行事のような長期一括入力は構造的に不可。
3. ADR-045 で **学校行事カレンダーのバックエンド（`school_calendar_sources` / `school_calendar_events`・tenant_isolation RLS・監査・取得 Job）は実装済み**だが、web 側の消費（表示・設定 UI・エディタ結線）が全て未実装のまま休眠している。

## 候補

### 保存先（長期行事データをどこに置くか）

| 候補 | 概要 | 評価 |
|---|---|---|
| **A. `school_calendar_events` 再利用（採用）** | ADR-045 の既存テーブルにファイル取込由来の行事を格納（`sourceId = null`・合成 `uid`） | RLS（tenant_isolation）・監査・`(school_id, startDate)` index が完成済み。**migration 不要**。iCal 取込と将来統合できる。`MAX_DRAFT_DAYS=7` を触らない |
| B. `daily_data` に複数日直書き | AI 取込の出力を各日の盤面データへ直接書く（7 日上限の緩和が必要） | 年間分で数百行の daily_data を生成。行事の後日修正で全行の整合を取る負担。7 日上限の意味（会話 AI の暴走防止）を壊す。不採用 |

### 盤面（サイネージ）への反映

| 候補 | 概要 | 評価 |
|---|---|---|
| **A. 第 1 段: エディタ確定のみ（採用）** | エディタに編集中日付の行事を表示し、教員がワンクリックで予定/連絡へ確定（基本時間割 seed のワンクリック確定と同じ UX 前例） | 教員が内容を確認してから盤面に出る＝AI 誤読の露出を防ぐ。表示スペースのデザイン判断が不要で最小 |
| B. 盤面自動表示まで一気に | 行事帯の新設 or `effective-daily-data` への読み取り時合成 | 教員工数ゼロだが、盤面レイアウト判断（パターン別）と誤読露出リスクが伴う。第 1 段の反応を見て再検討トリガで判断 |

### 取込の操作権限

| 候補 | 概要 | 評価 |
|---|---|---|
| **A. 教員 + 学校管理者（採用）** | エディタから教員も取込できる | フィードバック元が教員であり、手軽さが価値。学校単位データを教員が書ける点は取込履歴の監査（ルール1）で担保 |
| B. school_admin のみ | 学校の正データとして管理者画面に集約 | 安全だが教員の手軽さが落ちる。不採用 |

## 決定

**保存先 = `school_calendar_events` 再利用、盤面反映 = 第 1 段（エディタ確定のみ）、権限 = 教員 + 学校管理者**（いずれも 2026-07-11 ユーザー判断）。

1. **ファイル取込由来イベントの表現（migration 不要）**。`school_calendar_events` に `sourceId = null`・`uid = "file:<importBatchId>:<n>"`（`file:` プレフィクスで名前空間分離）で格納する。`raw` に `{ origin: "file-import", batchId, fileName, importedBy }` を保全する。**再取込は「`file:` 名前空間全体の置き換え」**（`source_id IS NULL AND uid LIKE 'file:%'` を削除して新バッチを挿入）を既定とする — 学校の年間行事表は 1 校 1 枚が通常形であり、バッチ単位の部分共存は複数ファイル運用が要件化してから（再検討トリガ）。

2. **★ iCal 掃除との衝突回避（必須の前提修正）**。`deleteStaleCalendarEvents`（`packages/db/src/queries/school-calendar.ts`）は現状 **school 単位で `keepUids` 外を全削除**するため、iCal ソース併用校では次回同期がファイル取込行事を誤削除する。削除条件に **`sourceId = <同期中のソース>` を追加**して掃除スコープをソース単位に絞る（`sourceId = null` のファイル取込行事は iCal 同期の掃除対象外になる）。この修正は取込機能より**先に** land する（スライス 1）。あわせて名前空間境界を両側からコードで強制する:
   - **iCal 取込側**: 外部フィードが `UID: file:...` を吐いても名前空間を侵食できないよう、iCal 由来 upsert は `file:` プレフィクスの UID を**リライト**（例 `ical:` を前置）または拒否する。
   - **ファイル削除側**: 置き換え削除の条件は uid プレフィクスだけに依存せず **`source_id IS NULL AND uid LIKE 'file:%'`** の二重条件とする（万一 `file:` uid が iCal 側に紛れても sourceId 非 null で保護される）。
   - ソース行削除で `onDelete: set null` になった **orphan iCal 行**（sourceId=null・非 `file:`）は上記いずれの掃除にも入らない。ソース削除時に配下イベントを同時削除するか orphan 掃除を追加するかは PR-A で決める（放置すると永久残留）。

3. **AI 構造化の専用経路（既存の 1 日分類とは別モード）**。既存の抽出レイヤ（exceljs / CSV / PDF / Gemini OCR）→ 既存のマスク済みパイプライン（PII soft-gate・監査、`runSectionDraft` の流儀）→ Gemini 構造化出力（Zod: `events[] { summary, startDate, endDate?, allDay, location? }`）。年間表は**年度文脈が必須**（「4/8」が何年か）なので、複数日下書きで実績のある「実在日付テーブルの system 注入」（`jstUpcomingDateTable`）の**年度版**（年度開始日〜翌 3 月の月/日↔曜日対応表）を注入し、モデルに暦算術をさせない。出力上限は `MAX_EVENTS_PER_SOURCE`（ADR-045 の 2000）と同水準でクランプし、切り捨ては件数を明示する（沈黙の切り捨て禁止）。

4. **保存前の確認 UI（必須）**。フォーマットが学校ごとに異なる以上 AI 誤読は残る。取込プレビュー（イベント一覧テーブル・行の修正/削除・年度の明示表示）で教員が確認してから保存する。**確認なしの自動保存はしない**。

5. **LLM 方針（ADR-045 決定 4 の追補・置換ではない）**。ADR-045 の「保存済み行事を LLM / embedding 経路に載せない」は**維持**する。本 ADR が加えるのは逆方向のみ: **教員がアップロードしたファイルは、既存のエディタ AI と同一のマスク済みパイプライン（PII soft-gate + 監査 + Vertex 同リージョン）を経由して AI 送信してよい**。アップロードファイルは教員が能動的に提供する半信頼入力であり、既存の `assistDraftAllFromFileAction` と同じ扱い。保存後の `school_calendar_events` は従来どおり非送信。

6. **権限と監査（ルール1 / ルール2）**。取込（INSERT/DELETE）は教員・school_admin の RLS セッション（tenant_isolation、school_id 一致）で実行し、`created_by / updated_by` = 実行者 uid。バッチ置き換え（前回取込の削除）も同セッションで `file:` 名前空間に限定して行う。iCal 由来（`sourceId` 非 null）の行事はエディタ/取込 UI から削除・上書きさせない。

7. **エディタ結線（第 1 段の可視化）**。エディタの編集中日付に該当する行事（`getCalendarEvents` の日付レンジ読み・RLS 委譲）を「この日の行事」として表示し、ワンクリックで予定（時刻付きは時限なし予定）/連絡へ確定挿入する。基本時間割 seed のワンクリック確定と同じ「AI/外部データは提案・教員が確定」の UX 原則。盤面への自動表示はしない（再検討トリガ）。

## 残存リスク

- ① **AI 誤読（日付ズレ・行事名の取り違え）**: 確認 UI（決定 4）で保存前に人間が見る。年度テーブル注入（決定 3）で暦算術ミスを構造的に減らす。
- ② **アップロードファイル内の PII**: 年間行事表に個人名（担当者等）が含まれうる。既存パイプラインの PII soft-gate / マスキングを通す（決定 5）。保存対象は行事名・日付・場所のみで、フリーテキスト全文は保存しない。
- ③ **教員による学校単位データの上書き**: 権限 A の代償。バッチ単位の置き換えに限定し、監査カラム + 取込履歴（`raw.batchId` / `importedBy`）で追跡可能にする。
- ④ **`file:` 名前空間の規約依存**: `origin` を専用カラムでなく uid プレフィクス + `raw` で表現するため、規約破りの uid が混ざると掃除・置き換えの境界が崩れる。決定 2 の二重条件（`source_id IS NULL` AND uid プレフィクス）と iCal 側リライトで両側から強制し、書き込み口をクエリ層のヘルパ（`upsertFileImportedEvents` 等）に単一化する。カラム化が必要になったら migration を追加（再検討トリガ）。
- ⑤ **「iCal 由来はエディタ/取込 UI から削除・上書きさせない」（決定 6）はアプリ層規約であり RLS では強制されない**（tenant_isolation はロール非依存の school_id 一致のみ）。PR-A の RLS テストは「教員セッションでも iCal 行を物理的には削除できてしまう」ことを前提に、アプリ層ヘルパの絞り込み（`source_id IS NULL` 限定の DELETE）をテスト期待値として固定する。

## 再検討トリガ

- 盤面への行事自動表示（第 2 段）が要件化した → パターン別の表示設計（行事帯 / `effective-daily-data` 合成）を別 ADR / design-ui で起こす。
- 取込ファイルが複数（学期別・学科別等）になり `file:` 単一名前空間で運用が回らなくなった → `origin` / `importBatchId` のカラム化 migration。
- iCal（ADR-045）とファイル取込の重複行事（同じ体育祭が両方から入る）が実運用で問題化した → 重複検知（同日・類似 summary）の導入。
- 年度をまたぐ取込（3 月に翌年度の表を読む）で年度推定が不安定 → 取込 UI で年度を明示選択させる（プレビューに年度セレクタ追加）。

## 影響

- **スライス計画（各 PR ≤500 行・ルール6）**:
  1. **PR-A（db レーン・schema-token）**: `deleteStaleCalendarEvents` のソーススコープ修正（決定 2）+ ファイル取込用クエリヘルパ（`upsertFileImportedEvents` / `replaceFileImportBatch` / 日付レンジ読みの再利用確認）+ RLS テスト（教員セッションの INSERT/DELETE 許可・他校拒否・iCal 行事の保護）。
  2. **PR-B（feature レーン）**: AI 構造化経路（Zod スキーマ・年度日付テーブル注入プロンプト・`packages/ai` or `apps/web/lib` の純ロジック + 単体テスト）。DB 書き込みなし。
  3. **PR-C（feature レーン）**: 取込 UI（エディタ/学校管理からのアップロード → プレビュー・修正 → 確定保存の Server Action）。
  4. **PR-D（feature レーン）**: エディタ「この日の行事」表示 + ワンクリック確定（決定 7）。
- スキーマ変更・Terraform 変更なし。新規外部通信なし（Vertex は既存経路）。
- `MAX_DRAFT_DAYS = 7`・会話 AI の挙動は不変。
- ADR-045 の決定 4 は本 ADR 決定 5 の限定追補を除き不変。
